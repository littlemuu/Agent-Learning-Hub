# Stage 2 学习笔记：Tool Use、RAG 和 Memory

本文件用于记录 Stage 2 的可复用知识点：搜索工具、RAG、chunk、embedding、检索结果引用、工具失败处理、短期上下文与长期记忆等。

当前状态：已完成最小 RAG 流程；下一步学习 chunk 边界和元数据设计。

## 最小 RAG 流程

RAG（检索增强生成）不是单独的知识库或向量数据库，而是一条把外部资料提供给 LLM 的流程：

```text
文档 -> chunk -> embedding -> 向量检索 -> 相关片段 -> LLM 基于片段回答并附引用
```

### 示例：学校规章问答

把 100 页学校规章做成问答系统时，可以按以下方式工作：

- `chunk`：按段落或语义把文档切成小片段，例如“请假规则”和“奖学金条件”。
- `embedding`：把每个片段转换为向量；同时保存原文、文档名、章节、页码或链接等元数据。
- 用户问“奖学金申请条件是什么？”时，也把问题转换为向量。
- 检索系统找出最相近的若干片段，并返回对应原文和元数据。
- LLM 根据问题和这些片段组织答案，同时标注规章章节、页码或链接。

### 每一步的输入与输出

1. `ingest`：原始文件（PDF、网页、Word 等） -> 可解析的文档文本和基础元数据。
2. `chunk`：文档文本 -> 按规则切分的片段及其来源元数据。
3. `embed`：片段文本 -> 片段向量，并将“向量 + 原文片段 + 元数据”写入向量存储。
4. `retrieve`：用户问题向量 -> 与问题最相关的原文片段及其元数据；不是只返回一串相近向量。
5. `answer with citations`：用户问题 + 相关片段 + 来源元数据 -> 基于证据的回答和可核查引用。

### 为什么不能把整本书作为一个 chunk

整本 100 页规章包含多个主题。一个向量需要同时表示所有主题，会稀释“奖学金条件”这一具体意图，导致相似度不够准确；即使检索到整本书，LLM 也需要在大量无关内容里寻找答案，成本更高且更容易遗漏或误答。

### 为什么向量存储必须保留原文和元数据

相似度检索只能找出相近的向量。回答阶段还需要从检索结果中取回原文片段，供 LLM 作为证据生成回答；同时需要页码、链接或文档标识等元数据，供用户核查引用。只保存向量时，LLM 看不到原文，系统也无法提供可核查来源。

## Chunk 边界与元数据

Chunk 应优先按完整语义单元切分，而不是机械地一条句子一个 chunk。内容短且主题单一时，同一主题的相邻句子应该保留在一起。例如“奖学金成绩条件”和“申请材料”共同构成奖学金申请规则，可作为一个 chunk；“请假审批流程”主题不同，应单独成块。

一个最小 chunk 元数据集合可以包含：

- `source`：源文件名或文档 URL；
- `page`：页码、章节或可定位位置；
- `chunk_id`：片段的唯一标识。

句子编号也可以作为更细的定位信息，但不能替代 `chunk_id`。当用户问“请假超过三天找谁审批？”时，应检索“请假审批流程”这个 chunk，而不是奖学金相关 chunk。

## Chunk 大小与 overlap

- Chunk 太小：检索定位可能更精确，但一条完整规则的条件、例外或结论可能被拆散，导致上下文不足。
- Chunk 太大：上下文较完整，但主题混杂，会稀释向量语义、增加无关内容和回答成本。
- Overlap：切分器主动把边界处的一小段内容复制到相邻 chunk 中，避免完整语义刚好被切断。

例如每个 chunk 为 300 个词、overlap 为 50 个词时，chunk 1 包含第 1-300 词，chunk 2 包含第 251-550 词。第 251-300 词是主动复制的边界上下文。

在“7 天内可退款；超过 7 天仅质量问题可人工审核”的规则里，不加 overlap 可能只检索到第一句，遗漏例外条件。Overlap 的代价是更多重复向量和 chunk，增加索引存储、检索去重与 LLM 上下文成本；因此只应保留足够连接语义的少量重叠。

## 检索质量与引用

检索结果不能直接全部交给 LLM 使用。至少需要三个控制点：

- `top_k`：最多保留多少个候选 chunk；
- `relevance threshold`：只使用相关度达到阈值的候选；
- 引用覆盖：每个关键结论都应能对应到具体 chunk 的来源。

对于“购买后 9 天，商品有质量问题，能退款吗？”的问题：

- “退款申请须在购买后 7 天内提交”是默认规则；
- “超过 7 天仅质量问题可人工审核”是例外规则；
- 两段必须一起引用，才能说明默认期限和例外的边界。

合格回答应说“可提交人工审核”，而不能说“保证退款”。当检索结果只有无关片段时，应说明没有找到相关资料或当前证据不足，必要时请求补充资料或调用其他工具；不能编造规则。

## 短期上下文、会话记忆与长期记忆

- 短期上下文：当前模型调用或当前对话窗口所需的信息，例如最近消息、刚返回的工具结果和刚检索出的文档片段。
- 会话记忆：同一任务或会话期间要持续保留的状态，例如本次已经访问的文件、已经执行过的命令或已发生的工具超时。
- 长期记忆：跨会话仍有价值、相对稳定且适合显式保存的信息，例如用户持续偏好中文。

判断时要先看保存范围和有效期。带有“今天”“明天”等时效的临时安排，不是长期记忆；它至多在当前会话内保留，到期后应丢弃。长期记忆还要考虑是否会过期、是否敏感、是否经过用户确认，不能把对话原样永久保存。

## 长期记忆的最小策略

长期记忆只应保存稳定、未来任务有用、来源明确的信息。一个最小记录可包含：`key`、`value`、`source`、`updated_at` 和可选的 `expires_at`。

- 显式语言偏好，例如“以后都用中文学习”，可以保存为长期记忆。
- 含有“今天”“明天”等相对时间的信息是临时状态，不应长期保存。
- 已由 `pyproject.toml` 等配置文件验证的项目约束可以保存，但它是可变的项目事实，不是用户偏好；配置变更时必须更新。
- 不应保存从聊天内容推测出的用户个人信息。未证实推断可能错误或敏感；应等待明确、相关且可验证的信息，而不是“先存再纠正”。

## 工具失败与空检索结果

工具失败后应先判断类型，再选择有上限的恢复动作：

- 临时失败（超时、网络波动）：记录原因后可有限重试；达到上限后可换备用工具或向用户报告无法完成验证。
- 确定性失败（缺少参数、权限不足、未知工具）：不要原样重试。先从当前任务中补齐或修正参数；只有无法确定必要参数时才追问用户。
- 空结果或低相关结果：可改写一次查询或扩大检索范围；若仍没有足够证据，应明确说明“没有找到可靠来源”，不能编造答案。

连续超时和空结果不同：连续超时意味着搜索没有成功完成，因此最终回复应说明尝试次数、超时状态和无法验证，而不是错误地声称“没有搜索到内容”。如果检索分数低于阈值，例如 `0.18`、`0.22` 低于 `0.60`，这些片段不能作为回答政策细节的证据。

当工具报缺少 `query` 时，如果用户请求已包含可确定的查询主题，Agent 应从请求中构造 `query` 并重试；只有无法确定查询内容时才追问用户。一次或有限次数重试仍超时后，合格回复应说明搜索未完成、暂时无法验证，而不是把超时误报成“没有结果”。

## 本地关键词检索练习

`codes/stage2_retrieval.py` 用固定的 chunk 模拟检索流程，不调用真实 embedding API。评分规则为：

```python
score = len(tokenize(query) & tokenize(chunk_text))
```

即查询词集合与 chunk 词集合的交集大小。`retrieve()` 对每个 chunk 打分，只保留 `score > 0` 的结果，保留 `id`、`text`、`source` 和 `score`，再按分数降序排列。

查询 `quality problem after 9 days` 时，`quality_exception` 以最高分排在第一位；查询 `zzzz` 时返回受控的“没有找到可靠资料”。这个练习说明了筛选、排序、证据保留和空结果处理，但关键词重合不等同于语义相似；后续真实 RAG 会用 embedding 相似度替换这一步。

## 关键词、embedding 与混合检索

关键词检索依赖字面词汇匹配，因此无法可靠理解 `faulty` 与 `quality problems`、`a week later` 与 `after 7 days` 这类改写或同义表达。Embedding 检索把文本编码为向量，更擅长找语义接近的片段，但不一定能可靠满足精确路径、文件名、错误码或版本号等硬约束。

- 查询 `policy.md#quality-exception` 这类精确标识时，优先关键词检索。
- 查询“商品坏了，一周后能退吗？”这类自然语言问题时，优先混合检索：用语义检索提高召回，再用关键词和规则保留精确证据。
- 混合检索不是把所有候选都交给模型；仍要过滤、排序、去重，并保留来源引用。

### 混合候选的保留规则

对自然语言查询 `faulty item one week later`，即使 `quality_exception` 没有字面关键词重合，只要语义分数达到阈值（例如 `0.93 >= 0.70`）也应保留。`refund_deadline` 的语义分数达到阈值，也应保留，因为它给出默认期限；`quality_exception` 则给出例外。两者共同决定完整回答边界。

候选规则可以写成：`lexical_score > 0 or semantic_score >= 0.70`。精确路径查询中，即使语义分数低于阈值，只要关键词命中就应保留，因为精确标识是强证据。
